# Tunny Dashboard 要件定義書

## 概要

Optunaを用いたブラックボックス最適化の結果を商用ツール水準で分析できる、ポストプロセッシング特化のブラウザ完結型ダッシュボードツール。

- **入力**: Optuna Journal形式（.log）＋アーティファクトディレクトリ（オプション）
- **対象規模**: 最大4目的関数 / 30変数 / 50,000サンプル（単目的にも対応）
- **技術スタック**: Rust/WASM + React/TypeScript + deck.gl + Zustand + Vite

---

## ユーザーストーリー

### ストーリー1: 多目的最適化結果の深い解析

- **である** 設計エンジニア **として**
- **私は** 最適化完了後にParetoフロントをインタラクティブに操作・絞り込みしたい
- **そうすることで** トレードオフを直感的に理解し、最終設計案を選定できる

### ストーリー2: 変数影響の把握

- **である** 解析担当者 **として**
- **私は** 30変数×4目的の感度ヒートマップとScatter Matrixを一覧したい
- **そうすることで** どの変数がどの目的関数に効くか全体像を把握できる

### ストーリー3: アルゴリズム比較

- **である** 最適化研究者 **として**
- **私は** 同一Journalファイル内の複数Studyを重畳・並列比較したい
- **そうすることで** サンプラーごとの性能差を客観的に評価できる

### ストーリー4: 最適化中のリアルタイム監視

- **である** 実験実行者 **として**
- **私は** 最適化実行中にブラウザを開いたまま試行追加を自動反映させたい
- **そうすることで** 良解が出たタイミングで手動停止判断ができる

### ストーリー5: 成果物レポートの共有

- **である** プロジェクトマネージャー **として**
- **私は** 分析結果を単一HTMLファイルとしてエクスポートしたい
- **そうすることで** Python環境のない相手にも分析結果を共有できる

## 機能要件（EARS記法）

### REQ-001: Journalファイル読み込み（通常要件）

- REQ-001: システムは Optuna Journal形式（.log）をブラウザのFile APIで読み込まなければならない
- REQ-002: システムは Optunaライブラリに依存せず、op_codeベースのステートマシンでJSONLを自前パースしなければならない
- REQ-003: システムは 入力を常に1つのJournalファイルとし、Journal内に複数のStudyが存在する場合はStudy選択UIで管理しなければならない
- REQ-004: システムは `create_study`・`create_trial`・`set_trial_param`・`set_trial_value`・`set_trial_state`・`set_trial_user_attr`・`set_trial_system_attr` の各op_codeを処理しなければならない
- REQ-005: システムは パース完了後、全データをWASMメモリ内のDataFrameとして常駐させなければならない

### REQ-010: データ変換・前処理（通常要件）

- REQ-010: システムは `FloatDistribution`・`IntDistribution`・`CategoricalDistribution`・`UniformDistribution` の内部スケール値を表示値に逆変換しなければならない
- REQ-011: システムは `RUNNING`・`PRUNED`・`FAIL` などの不完全なトライアルを除外または保留リストで管理しなければならない
- REQ-012: システムは `set_trial_user_attr` の数値型をfloat64列、文字列型をカテゴリ列としてDataFrameに追加しなければならない
- REQ-013: システムは `set_trial_system_attr` の`constraints`を個別列（c1, c2, c3...）に展開し、`is_feasible` フラグ列と `constraint_sum` 列を派生列として追加しなければならない
- REQ-014: システムは GPUバッファとして `positions (Float32Array)`・`colors (Float32Array)`・`sizes (Float32Array)` を保持しなければならない
- REQ-015: フィルタ操作時、システムは `colors[i*4+3]` （alpha値）のみを更新し、positions/sizesの変更は行ってはならない

### REQ-020: Study選択・モード切り替え（条件付き要件）

- REQ-020: Journalファイル読み込み完了時、システムは 全Studyの一覧（study_id・名前・目的数・試行数）を表示しなければならない
- REQ-021: COMPLETE試行数が最多のStudyをデフォルト選択として、システムは 提示しなければならない

- REQ-023: ユーザーが2〜4目的のStudyを選択した場合、システムは 多目的モード（Pareto中心レイアウト）に切り替えなければならない
- REQ-024: Study切り替え時、システムは フィルタ・選択状態をリセットしなければならない（変数名が一致する場合は引き継ぎオプションを提供してよい）

### REQ-030: 画面レイアウト（通常要件）

- REQ-030: システムは ToolBar・Left Panel（240px）・Main Canvas・Bottom Panel の4エリアで構成されなければならない

- REQ-032: システムは 多目的解析モード（Mode A）・変数空間探索モード（Mode B）・収束分析モード（Mode C）・フリーレイアウト（Mode D）の4レイアウトモードを提供しなければならない

## 非機能要件

### パフォーマンス（NFR-001〜）

- NFR-001: 5万サンプル×30変数×4目的の規模で快適に動作しなければならない（WebGLレンダリングを強制）
- NFR-002: WASMの`filter_by_ranges()`は50,000件で5ms以内に完了しなければならない
- NFR-003: GPU alpha値の更新は1ms以内/グラフとしなければならない
- NFR-004: 初期Journalパース（50,000行）は5秒以内に完了しなければならない
- NFR-005: Trade-off Navigatorのスコアリングは50,000点で1ms以内としなければならない
- NFR-006: k-meansクラスタリング（50,000点）はPCA〜完了まで400ms以内としなければならない
- NFR-007: HTMLレポート生成は1秒以内としなければならない
- NFR-008: 差分ライブ更新（1,000行）の合計処理時間は100ms以内としなければならない

### アーキテクチャ（NFR-010〜）

- NFR-010: システムはサーバー不要・単一HTMLファイルで動作しなければならない（ブラウザ完結）
- NFR-011: Rust/WASMコアは `rust_core` として、WASMターゲットとNativeターゲットの両方にビルドできる構造としなければならない
- NFR-012: WASMのメモリ空間にDataFrame全体を保持し、フィルタ操作ごとのJSへのデータ転送を発生させてはならない
- NFR-013: 状態管理にはZustandを使用し、Brushing状態は`selectedIndices`・`filterRanges`・`highlighted`・`colorMode`の4プロパティで管理しなければならない

### セキュリティ（NFR-020〜）

- NFR-020: Journalファイルはブラウザ外に送信してはならない（完全ローカル処理）
- NFR-021: File System Access APIを使用する際は ユーザーの明示的な許可を必要としなければならない

### ユーザビリティ（NFR-030〜）

- NFR-030: カラーリングモードは Paretoランク・目的関数値・試行番号・クラスタIDの4種類から選択可能でなければならない
- NFR-031: フリーレイアウト（Mode D）では 4×4グリッドスナップ方式でドラッグ&ドロップ配置を提供しなければならない
- NFR-032: レイアウト設定はJSONで保存・読み込み可能でなければならない

- EDGE-002: 分散最適化時の同一trial_idへの複数書き込みに対応しなければならない
- EDGE-003: 全てのトライアルが不完全（FAIL/RUNNING）な場合は、空のDataFrameとして表示しエラーメッセージを出力しなければならない
- EDGE-004: File System Access API非対応ブラウザでライブ更新を有効化しようとした場合は、手動更新ボタンへのフォールバックを案内しなければならない

### 境界値

- EDGE-101: 1目的（単目的）のJournalでPareto関連の操作を行った場合は、無効操作として無視しなければならない
- EDGE-102: 変数が0個または目的が0個のStudyを選択した場合は、対応する可視化を非表示にしなければならない
- EDGE-103: 50,000点のSVGエクスポートは非現実的なため、自動的に上位5,000点に絞り込み警告メッセージを表示しなければならない
- EDGE-104: アーティファクトディレクトリが指定されていない場合は、アーティファクト関連UIを非表示にしなければならない
- EDGE-105: .onnxファイルなし・shap_values.jsonなしの場合、PDPは線形モデル（WASM）でフォールバックし、その旨を案内しなければならない
- EDGE-106: クラスタ数kを1に設定した場合は、クラスタリング実行を拒否しエラーメッセージを表示しなければならない
- EDGE-107: 選択サンプルが0件のときに再クラスタリングを実行した場合は、全サンプルを対象として実行しなければならない

## 受け入れ基準

### 機能テスト

- [ ] 50,000試行のJournalファイルを5秒以内に読み込みDataFrameを構築できる
- [ ] 変数フィルタスライダー操作から全グラフ更新まで100ms以内（体感的に即時）
- [ ] ParetoフロントをBrushで選択すると、Bottom Tableとカウンタが連動更新される
- [ ] 単目的StudyではPareto関連UIが非表示になり、収束分析レイアウトが表示される
- [ ] 同一Journal内の2つのStudyを選択して重畳比較ができる
- [ ] k-meansクラスタリング（k=4、目的空間）が400ms以内に完了する
- [ ] Spearman感度ヒートマップ（30変数×4目的）が500ms以内に表示される
- [ ] スタンドアロンHTMLレポートを1秒以内に生成しダウンロードできる
- [ ] ライブ更新中に新規試行が追加されても、Brushingと視点が変更されない
- [ ] アーティファクトディレクトリ指定後、Pareto点ホバーで画像サムネイルが表示される

### 非機能テスト

- [ ] ブラウザ（Chrome/Edge）で追加インストールなしに動作する
- [ ] Journalファイルがブラウザ外のネットワークに送信されない（開発者ツールで確認）
- [ ] 50,000点のScatter Plotが60fps（16.7ms/frame）でインタラクティブに操作できる
- [ ] 単一HTMLファイルとして配布でき、Pythonなしで動作する
