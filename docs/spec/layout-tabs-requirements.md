# layout-tabs 要件定義書

## 概要

現状のレイアウト切り替えは ToolBar 上の A〜D ボタンと FreeLayoutCanvas 内のプリセットボタンという2つの独立したコントロールが存在し、操作が直感的でない。本機能は、これらをタブ型 UI に統合して、ユーザーがどのレイアウトが現在アクティブかを一目で把握し、簡単に切り替えられるようにする。

## 現状の問題点

1. **不透明なラベル**: A/B/C/D というラベルは各レイアウトの意味を表現していない
2. **二重コントロール問題**: ToolBar の「A〜D ボタン」（`setLayoutMode`）と FreeLayoutCanvas 内の「プリセットボタン A〜C」（`setFreeModeLayout`）が別々に存在し、それぞれが異なるアクションを呼び出す
3. **アクティブ状態の不明確さ**: 現在どのモードが選択されているかがボタンの外観から判別しにくい

## ユーザーストーリー

### ストーリー1: レイアウト切り替え

- **である** 最適化結果を分析するユーザー **として**
- **私は** 現在のレイアウトがどれかを一目で確認し、クリック1回で他のレイアウトに切り替えたい
- **そうすることで** 分析状況に応じたグラフ配置を素早く選択できる

### ストーリー2: レイアウトの意味理解

- **である** 初めてダッシュボードを使うユーザー **として**
- **私は** 各レイアウトオプションが何を表示するかをラベルから推測したい
- **そうすることで** 試行錯誤なく目的のレイアウトを選べる

## 機能要件（EARS記法）

### 通常要件

- REQ-001: システムは ToolBar にタブ型レイアウト切替コンポーネントを表示しなければならない
- REQ-002: システムは各タブに日本語の説明的ラベルを表示しなければならない（A→「4分割」、B→「左大」、C→「縦並び」、D→「フリー」）
- REQ-003: システムは FreeLayoutCanvas 内のプリセットボタン（A/B/C）を除去しなければならない

### 条件付き要件

- REQ-101: ユーザーがタブ「4分割」をクリックした場合、システムは `setLayoutMode('A')` を呼び出し、かつ Mode A に対応するプリセットレイアウトを `setFreeModeLayout` で適用しなければならない
- REQ-102: ユーザーがタブ「左大」をクリックした場合、システムは `setLayoutMode('B')` を呼び出し、かつ Mode B に対応するプリセットレイアウトを `setFreeModeLayout` で適用しなければならない
- REQ-103: ユーザーがタブ「縦並び」をクリックした場合、システムは `setLayoutMode('C')` を呼び出し、かつ Mode C に対応するプリセットレイアウトを `setFreeModeLayout` で適用しなければならない
- REQ-104: ユーザーがタブ「フリー」をクリックした場合、システムは `setLayoutMode('D')` を呼び出さなければならない（`freeModeLayout` は変更しない）
- REQ-105: 現在の `layoutMode` に対応するタブは、他のタブと視覚的に区別されるアクティブスタイルで表示されなければならない
- REQ-106: アクティブなタブに再度クリックしても、システムはモードの変更や再適用を行わないこと（べき等）

### 状態要件

- REQ-201: レイアウトモードが 'A'〜'D' のいずれかである場合、対応するタブがアクティブ状態で表示されなければならない

### 制約要件

- REQ-401: タブコンポーネントは Tailwind CSS クラスを使用してはならない（インラインスタイルのみ）
- REQ-402: `useLayoutStore` の `layoutMode` / `setLayoutMode` のシグネチャは変更してはならない
- REQ-403: タブコンポーネントは `data-testid="layout-tab-{mode}"` 属性を持たなければならない（`layout-tab-A` 〜 `layout-tab-D`）
- REQ-404: `LayoutTabBar` コンポーネントは `frontend/src/components/layout/LayoutTabBar.tsx` に配置しなければならない
- REQ-405: 既存の `data-testid="layout-btn-A"` 〜 `data-testid="layout-btn-D"` は削除すること（テストも更新）

## UI/UX 要件

### タブ外観

- REQ-501: タブバーは ToolBar の右端エリアに水平並びで配置すること（既存の「ライブ開始/停止」ボタンの左隣）
- REQ-502: アクティブタブのスタイル: `background: var(--accent)`, `color: #fff`, `fontWeight: 700`, ボトムボーダーなし
- REQ-503: 非アクティブタブのスタイル: `background: var(--bg)`, `color: var(--text-muted)`, `border: 1px solid var(--border)`, ホバー時 `background: var(--hover)`
- REQ-504: タブ全体の外観: 上部に `2px solid var(--border)` のアンダーライン（タブバー下辺）を持つコンテナ内に配置
- REQ-505: フォントサイズ: `13px`、パディング: `4px 14px`
- REQ-506: タブの並び順: 4分割 → 左大 → 縦並び → フリー（左から右）

### インタラクション

- REQ-601: タブ切り替えはマウスクリック1回で完了すること
- REQ-602: プリセット適用時の `window.confirm` ダイアログは廃止すること（タブクリック = 即時適用）

## 非機能要件

### パフォーマンス

- NFR-001: タブ切り替えによる再レンダリングは 16ms 以内（60fps）に完了すること

### テスト容易性

- NFR-101: `LayoutTabBar` は `useLayoutStore` を直接購読するため、テスト時は layoutStore をモックすること

## Edgeケース

### エラー処理

- EDGE-001: `freeModeLayout` が `null` の状態でプリセットタブ（A/B/C）をクリックした場合、プリセットレイアウトが正しく適用されること（`null` チェック不要 — `setFreeModeLayout` 自体が null を受け入れる）

### 境界値

- EDGE-101: 存在しない layoutMode（型安全で防ぐ）がストアに設定されている場合、どのタブもアクティブにならないこと（現実的には発生しないが、TypeScript で型ガード）

## 受け入れ基準

### 機能テスト

- [ ] タブ「4分割」クリックで `layoutMode === 'A'` かつ Pareto Front が左上に配置される
- [ ] タブ「左大」クリックで `layoutMode === 'B'` かつ 3チャートレイアウトが適用される
- [ ] タブ「縦並び」クリックで `layoutMode === 'C'` かつ 3チャートレイアウト（下部横長）が適用される
- [ ] タブ「フリー」クリックで `layoutMode === 'D'` かつ `freeModeLayout` は変更されない
- [ ] アクティブタブが `layoutMode` と一致して視覚的に区別される
- [ ] FreeLayoutCanvas 内のプリセットボタンが削除されている

### UI テスト

- [ ] `data-testid="layout-tab-A"` 〜 `data-testid="layout-tab-D"` が DOM に存在する
- [ ] アクティブタブに `aria-selected="true"` または相当する属性が設定される
- [ ] 既存の `layout-btn-A` 〜 `layout-btn-D` が DOM に存在しない

### 退行テスト

- [ ] ToolBar の既存機能（ファイル入力、Study セレクタ、ライブ更新ボタン）が正常動作する
- [ ] FreeLayoutCanvas のドラッグ&ドロップ・チャート削除が正常動作する
