## 1. プロジェクト概要

### 目的

Optunaを用いたブラックボックス最適化の結果を、商用ツール水準の利便性で分析できるダッシュボードツールを提供する。OptunaDashboardが「最適化実行中のリアルタイム監視」に強い一方、本ツールは**最適化完了後の深い解析に特化**したポストプロセッシング分析ツールとして位置づける。

### 設計方針

- Optunaライブラリには依存せず、Journal形式を自前パーサで直接読み込む
- **入力は常に1つのJournalファイル**。Journal内に複数のStudyが存在する場合はStudy選択UIで管理する
- 複数Journalの同時読み込みは対象外とする
- 5万サンプル×30変数×4目的の規模で快適に動作すること（単目的も対応）
- Brushing & Linkingによる全グラフ連動インタラクションを核心機能とする
- 配布の容易さを重視し、サーバー不要・単一ファイルで動作することを目指す

## 2. OptunaDashboardとの差分

### OptunaDashboardが持つ機能（現状）

| 機能カテゴリ | 内容 |
|---|---|
| 可視化 | 最適化履歴グラフ、コンタープロット、パラメータ重要度、Parallel Coordinates、Pareto Front（多目的）、中間値可視化 |
| LLM統合（v4.6〜） | 自然言語によるトライアルフィルタリング、自然言語プロンプトからのPlotlyグラフ自動生成 |
| 動作環境 | Python版、SQLite3+Rust/WASMのブラウザ完結版、JupyterLab拡張、VSCode拡張 |
| インタラクション | フォームウィジェット（Choice/Slider/TextInput）による人間フィードバック、Preferential Optimization |
| 既知の制限 | コンタープロットグリッド（全パラメータペア一覧）は未実装 |

### 本ツールの差別化ポイント

#### 速度面（最大の優位点）

| 課題 | OptunaDashboardの現状 | 本ツールの解決策 |
|---|---|---|
| 大規模データ描画 | PlotlyベースでDOM描画、5万点は重い | WebGL強制レンダリング、全点高速表示 |
| フィルタ応答速度 | Python側で再計算→再送信のラウンドトリップ | WASM内でクライアント完結処理 |
| 初期ロード | 全データを一括取得 | Progressive Loading、差分ストリーミング |
| Parallel Coordinatesの軸数 | 多軸（30変数）では操作が重くなる | WebGL実装による30軸の快適操作 |

#### 分析機能面（差別化領域）

**多目的解析**
- Brushing & Linking: ある図での選択が全図に即時連動（OptunaDashboardは図ごとに独立）
- 4目的のParetoフロント可視化: 3D+色軸での4次元表現
- Hypervolume推移の詳細分析: 世代ごとの寄与度分解
- Trade-off Navigator: Paretoフロント上をスライダーで移動しながらトレードオフを探索

**変数空間の分析**
- Scatter Matrix（全変数ペア）: 30変数×30変数の一覧表示（OptunaDashboardは2変数ずつのコンタープロットのみ）
- Partial Dependence Plot（PDP）: 変数と目的関数の関係をサロゲートで補間可視化
- クラスタリング表示: 設計空間をk-means等でグループ化し色分け
- 設計変数の分布変化: 試行番号の進行による探索分布の推移

**感度・重要度分析**
- 目的関数ごとの個別重要度: 4目的それぞれに対して変数がどう影響するかを分離表示
- 感度分析ヒートマップ: 変数×目的関数の寄与行列

**UI機能**
- フィルタ条件の保存・再現
- 複数Study比較（Paretoフロントや分布を重畳表示）
- Brushingで選択した点のCSV出力
- レイアウト設定の保存・読み込み
- 選択図+サマリーのHTMLエクスポート

### 差別化ポジション

```
OptunaDashboard   →  "最適化実行中のリアルタイム監視"に強い
本ツール          →  "最適化完了後の深い解析"に特化
```

## 4. データ設計

### Optuna Journal形式の構造

Journal形式（`.log`）は1行1JSONのAppend-onlyログ。

```jsonl
{"op_code": "create_study", "study_id": 0, "directions": ["minimize","minimize"], ...}
{"op_code": "create_trial", "study_id": 0, "trial_id": 0, ...}
{"op_code": "set_trial_param", "trial_id": 0, "param_name": "x1", "param_value": 3.14, ...}
{"op_code": "set_trial_value", "trial_id": 0, "value_idx": 0, "value": 0.923, ...}
{"op_code": "set_trial_state", "trial_id": 0, "state": "COMPLETE", ...}
```

### Journalパーサの設計方針

Optunaライブラリを使わず、`op_code`ベースのステートマシンとして実装する。

```
Journalファイル読み込み
    ↓
行ごとにJSONパース
    ↓
op_codeで分岐処理
 ├─ create_study        → メタデータ構築
 ├─ create_trial        → トライアル初期化
 ├─ set_trial_param     → 変数値の蓄積
 ├─ set_trial_value     → 目的関数値の蓄積
 ├─ set_trial_state     → ステータス確定
 ├─ set_trial_user_attr → user_attrs の蓄積  ← 追加
 └─ set_trial_system_attr
      key="constraints" → 制約違反量の蓄積  ← 追加
      key="artifacts"   → アーティファクトメタデータ
    ↓
DataFrame（WASM Memory内）に変換
```

### user_attrsのパースと統合

```jsonl
{"op_code": "set_trial_user_attr",
 "trial_id": 42, "key": "constraint_violation", "value": 0.023}
{"op_code": "set_trial_user_attr",
 "trial_id": 42, "key": "solver_status", "value": "converged"}
```

```
パース方針:
  数値型  → float64列として追加
  文字列型 → カテゴリ列として追加
  リスト・dict型 → JSON文字列として保持

列名の衝突回避:
  変数名と重複する場合 → "user_attr::{key}" でプレフィックス付与
```

### 制約条件のパースと統合

```jsonl
{"op_code": "set_trial_system_attr",
 "trial_id": 42, "key": "constraints",
 "value": [0.0, -0.023, 0.015]}
```

```
パース方針:
  各要素を c1, c2, c3 ... として個別列に展開
  値 ≤ 0 = 制約満足、値 > 0 = 制約違反
  "is_feasible" フラグ列を derived column として追加
    is_feasible = (全制約値 ≤ 0) の論理AND
  "constraint_sum" 列も追加（全違反量の合計、デバッグ用）
```

### 実装上の注意点

- `RUNNING`・`PRUNED`・`FAIL`など**不完全なトライアルの扱い**を明示的に設計する
- 分散最適化時に同一trial_idへの複数書き込みが発生しうる
- `param_value`は内部スケール値で格納されており、`distribution`フィールドを使った逆変換が必要

### 分布の逆変換マッピング

| distribution型 | 内部値 | 表示値への変換 |
|---|---|---|
| `FloatDistribution` | float | そのまま（log=Trueなら`exp`） |
| `IntDistribution` | float | `round` → int |
| `CategoricalDistribution` | int (index) | `choices[index]` |
| `UniformDistribution` | float | そのまま（旧形式互換） |

### GPUバッファ設計

5万点の高速描画のため、データをTypedArrayとしてGPUバッファに保持する。

```
positions: Float32Array  // x, y座標 (50000 × 2)
colors:    Float32Array  // r, g, b, a (50000 × 4)
sizes:     Float32Array  // 点サイズ  (50000 × 1)

フィルタ時の更新: colors[i*4+3] = 0.0 (alpha=0で非表示)
→ positions/sizesは変更不要 → GPU転送量が最小
```

## 6. Brushing & Linking 状態管理設計

### 核心概念

「選択状態という単一の真実（Single Source of Truth）を全グラフが購読する」設計。

### Storeの設計（Zustand）

```typescript
interface SelectionStore {
  // 状態
  selectedIndices: Uint32Array;        // 選択されたtrial_idの集合
  filterRanges: Record<string, Range>; // 軸ごとのフィルタ範囲
  highlighted: number | null;          // ハイライト中のtrial_id
  colorMode: 'objective' | 'cluster' | 'rank' | 'generation';

  // アクション
  brushSelect(indices: Uint32Array): void;
  addAxisFilter(axis: string, min: number, max: number): void;
  removeAxisFilter(axis: string): void;
  clearSelection(): void;
  setHighlight(index: number | null): void;
}
```

### 選択モード

**Mode 1: Brush Selection（範囲選択）**
- 操作: グラフ上をドラッグ
- 結果: selectedIndicesに範囲内のtrialが追加
- 実装: 矩形/多角形とFloat32Array座標の交差判定

**Mode 2: Axis Filter（軸フィルタ）**
- 操作: PCPの軸上でブラシをドラッグ
- 結果: filterRangesが更新 → selectedIndicesが再計算
- 実装: WASMで範囲クエリ → インデックス返却

**Mode 3: Click Highlight（単点ハイライト）**
- 操作: 点をクリック
- 結果: highlightedIndexが設定 → 詳細パネル表示
- 実装: GPU pick buffer（オフスクリーンレンダリング）で点特定

### 更新の高速化（重要）

```
【ナイーブな実装（遅い）】
軸フィルタ変更 → Store更新 → 全グラフReact re-render
→ 全グラフがフルデータで再計算 → 描画（100ms〜数秒）

【正しい実装（速い）】
軸フィルタ変更
→ WASM filter_by_ranges()（< 5ms）
→ selectedIndices (Uint32Array) のみ更新
→ 各グラフはGPUバッファのalpha値のみ書き換え（< 1ms）
→ requestAnimationFrameで次フレーム描画
```

**ポイント**: グラフコンポーネントをReactの再レンダリングサイクルから切り離す。WebGLの描画ループは独自のrAFループで回し、StoreのsubscribeでDOMを経由させずに直接受け取る。

## 7. WebGL描画技術選定

### 可視化タイプ別ライブラリ

| 可視化 | 採用技術 | 理由 |
|---|---|---|
| Scatter Plot / Pareto 3D | deck.gl ScatterplotLayer / PointCloudLayer | 5万点をGPUバッファで一括描画、色・サイズの動的更新が高速 |
| Parallel Coordinates (30軸) | regl ベース自前実装 または ECharts WebGLモード | PlotlyのPCPは軸数増加で著しく重くなる |
| Scatter Matrix (30×30) | OffscreenCanvas + WebWorker | 表示範囲内セルのみ遅延レンダリング |
| ヒストグラム / 分布 | ECharts | 軽量で十分 |

### WebGL / WASM 連携

```
Rust WASM側                    JS / WebGL側
─────────────────              ──────────────────────
filter_by_ranges()    →        SharedArrayBuffer経由
DataFrame常駐                  （コピーなし転送）
インデックス配列生成   →        GPU Buffer更新
統計計算              →        グラフ描画
```

WASMのメモリ空間にDataFrame全体を保持し、フィルタ操作のたびにJSへのデータ転送を発生させないことが速度の鍵。

### カラースケール設計（全ビュー共通）

| モード | 内容 |
|---|---|
| Paretoランク | Rank1=赤、Rank2=橙、Rank3=黄、非Pareto=灰 |
| 目的関数値 | 選択目的のグラデーション（minimize: 青=良、赤=悪） |
| 試行番号 | 初期=青、後期=赤（探索の進化を可視化） |
| クラスタ | k-means等グループごとに固有色 |

## 8. Scatter Matrix設計（30×30対応）

### 問題の規模感

30変数 + 4目的 = 34軸の全ペア数:

| ペア種別 | セル数 |
|---|---|
| 変数×変数 | 30×29/2 = 435 セル |
| 変数×目的 | 30×4 = 120 セル |
| 目的×目的 | 4×3/2 = 6 セル |
| **合計** | **561 セル** |

全セル同時描画は非現実的なため、段階的レンダリング戦略を採用。

### 3段階レンダリング戦略

**Stage 1: サムネイル表示（初期表示、即時）**
- 全34×34セルを低解像度で一覧
- 各セル 80×80px、点は最大500点にダウンサンプリング
- 目的: 全体の相関構造を素早く把握

**Stage 2: ホバー時の拡大プレビュー**
- マウスオーバーしたセルを 300×300px にポップアップ
- 全5万点をWebGLで描画
- 目的: 気になるセルを素早く確認

**Stage 3: クリックでフルサイズ展開**
- 選択セルをメインキャンバスに 600×600px で展開
- Brushing & Linking完全有効
- 目的: 詳細な分析・選択操作

### レンダリング優先順位と時間目標

```
Priority 1（T+3〜4秒）: Mode 2 必須セル（変数×目的 30×4 = 120セル）
  → 最も頻繁に使われる構成を最優先
  → 目標: Journalパース完了後1秒以内

Priority 2（T+4〜5秒）: 対角ヒストグラム（34セル）
  → 各変数・目的の分布把握に重要
  → 目標: Priority 1完了後1秒以内

Priority 3（T+5秒〜）: 変数×変数（435セル）
  → ビューポートに近いセルをIntersection Observerで優先
  → 表示範囲外のセルは遅延

Priority 4（オンデマンド）: ホバー時拡大（300×300px全点描画）
  → ユーザーがホバーしたときのみ実行
```

### WebWorker並列処理の設計

```
Worker分割方針: 行グループ単位

Worker 1: 行 0〜9  （変数 x1〜x10）
Worker 2: 行 10〜19 （変数 x11〜x20）
Worker 3: 行 20〜29 （変数 x21〜x30）
Worker 4: 行 30〜33 （目的・対角）

各Workerが担当行のサムネイルを独立して計算
完了次第メインスレッドに転送 → OffscreenCanvasに描画

目標:
  120セル（Mode 2）: 4 Worker並列で 〜500ms
  全1,156セル:       〜2秒
```

### GPUタイル描画

```
1枚のOffscreenCanvasに全セルをタイル状に描画
WebGLのinstanced drawingで全セルを1ドローコールで描画
→ セル数×点数分のインスタンスを一括GPU転送
```

### 表示モード

**Mode 1: 変数×変数（設計空間の相関確認）**
- 34×34の下三角行列
- 対角: ヒストグラム（変数の分布）
- 下三角: 散布図
- 上三角: 相関係数ヒートマップ（数値のみ・軽量）

**Mode 2: 変数×目的（感度の全体把握）**
- 30行×4列 の矩形マトリクス
- 最も頻繁に使われる構成
- どの変数がどの目的に効くか一目で把握

**Mode 3: カスタム選択**
- 表示する変数・目的をユーザーが選択
- 重要度上位N変数だけ表示など

### ダウンサンプリング戦略（サムネイル用）

単純ランダムサンプリングではPareto付近の点が消えてしまうため:

```
Step 1: Pareto点は必ず含める（例: 200点）
Step 2: 残り300点を空間的均等サンプリング
         → グリッド分割してグリッドごとに1点選択
Step 3: selectedIndicesに含まれる点は必ず含める
         → Brushing後のサムネイルでも選択状態を反映
実装: Rust/WASMで高速実行（< 1ms）
```

### 軸ソートオプション

- 目的関数との相関係数順（最重要変数が上）
- パラメータ重要度（fANOVA）順
- ユーザー定義グループ順
- 変数名アルファベット順

## 10. クラスタリング機能設計

### 目的と位置づけ

クラスタリングは単なる色分けではなく、**設計者が気づいていない解の構造を発見する**分析手段。

```
クラスタリングで答えたい問い:
  ① 設計空間にどんなグループが存在するか？
  ② 各グループはどんな変数パターンを持つか？
  ③ Paretoフロント上のトレードオフはグループと対応しているか？
  ④ 特定のクラスタだけに絞って深掘りしたいとき
```

### クラスタリング対象空間

| モード | 入力 | 用途 |
|---|---|---|
| **Mode A: 目的空間**（デフォルト） | obj1〜obj4 | トレードオフのパターン分類 |
| **Mode B: 変数空間** | x1〜x30 | 設計パラメータの類似グループ発見 |
| **Mode C: 結合空間** | x1〜x30 + obj1〜obj4（正規化後） | 変数と性能が共に似た解のグループ発見 |

### アルゴリズム選定

| アルゴリズム | 速度 | クラスタ形状 | ノイズ耐性 | k指定 | 50000点での実用性 |
|---|---|---|---|---|---|
| **k-means** | ◎ | 球状のみ | △ | 必要 | ◎（主軸） |
| **Mini-batch k-means** | ◎ | 球状のみ | △ | 必要 | ◎ |
| **HDBSCAN** | △ | 任意形状 | ◎ | 不要 | △（オプション） |
| **階層クラスタリング** | ✕ | 任意 | ◯ | 不要 | ✕（O(n²)） |

**主軸: k-means**、オプションで **HDBSCAN**（Paretoフロントの構造発見向け）。

#### 高次元対処フロー

```
生データ（30〜34次元）
    ↓
次元削減（UMAP or PCA）→ 2〜5次元
    ↓
クラスタリング（k-means / HDBSCAN）
    ↓
クラスタラベルを元の全次元データに付与
    ↓
全ての可視化でクラスタ色を使用
```

次元削減はクラスタリングの**前処理のみ**に使用し、可視化には常に元の変数・目的値を使う。

### 次元削減技術

| 手法 | 速度 | 局所構造 | 大域構造 | 用途 |
|---|---|---|---|---|
| **PCA** | ◎ | △ | ◎ | 前処理として常時適用 |
| **UMAP** | ◯ | ◎ | ◯ | オプション・構造発見用 |
| **t-SNE** | △ | ◎ | ✕ | 非推奨（大域構造を失う） |

UMAPの50,000点での計算時間は数十秒になりうるため、**バックグラウンドWorkerで非同期実行**が必須。

### クラスタ数の自動推定

```
Elbow法:
  k=2〜15 でk-meansを実行
  各kのWCSS（within-cluster sum of squares）をプロット
  「肘」の位置を自動検出（2次微分の最大点）
  → 「k=4を推奨します」と表示、ユーザーが変更可能

Silhouette法:
  各kのSilhouette係数を計算
  係数が最大のkを推薦
```

#### UMAP 2D可視化（オプション）

### Brushing & Linkingとの統合

```
操作パターン1: クラスタ → ブラシ
  クラスタ一覧でC2をクリック
  → selectedIndices = C2に属するtrial_idの集合
  → 全グラフでC2のみ明るく表示

操作パターン2: ブラシ → クラスタ確認
  Pareto図で範囲選択
  → クラスタ一覧に「選択中: C1=45%, C2=38%, C3=17%」
    をリアルタイム表示

操作パターン3: 選択サンプルのみ再クラスタリング
  Brushingで絞り込み後に再実行
  → 絞り込んだ領域内の細かい構造を発見
```

### 計算分担と時間目標

```
┌─────────────────────────────────────────────┐

│  - PCA（固有値分解）           〜  50ms      │
│  - k-means（Lloyd's algorithm）〜 200ms      │
│  - Silhouette係数・WCSS計算    〜 100ms      │
│  - クラスタ統計量（mean, std）              │
│  - t検定（クラスタ間有意差判定）            │
│  合計目標: 〜 400ms 以内                    │
├─────────────────────────────────────────────┤
│  WebWorker (JS)                             │
│  - UMAP計算（umap-js）         5〜30秒      │
│  ※ RustのWASM実装が未成熟なためJSで対応     │
└─────────────────────────────────────────────┘
```

## 12. Partial Dependence Plot（PDP）設計

### 位置づけと目的

PDPはScatter MatrixやParameter Importanceが「何が重要か」を示すのに対し、**「どう変化するか」を定量的に示す**機能。サロゲートモデル（代理モデル）を使って実サンプルが存在しない領域も補間し、変数と目的の関係を可視化する。

```
Scatter Matrix          PDP
──────────────          ──────────────────────────
実際のサンプル点を表示  → サンプルがない領域も推定
相関の有無を確認        → 変数と目的の関係を定量化
「x5が大きいとobj1が    → 「x5を3.0→7.0に変えると
 小さい傾向がある」       obj1は平均0.3改善する」
```

### Python依存方針：段階的フォールバック設計

**PDP以外の全機能はPython不要**。PDPのみ精度レベルに応じて以下の2段階で動作する。

```
.onnxファイルあり？
    ↓ Yes                        ↓ No
高精度PDP                    簡易PDP（WASM内線形モデル）
（ONNX Runtime Web）         + 案内メッセージを表示
Random Forest 相当
```

| 状況 | 動作 | Python必要？ |
|---|---|---|
| .onnxなし | 線形モデルによる簡易PDP | **不要** |
| .onnxあり | Random Forestによる高精度PDP | 生成時のみ1回 |
| Python環境なし | 簡易PDPのみ（他機能は全て動作） | **不要** |

#### 未読み込み時のUI

```

│                                          │
│ ⚠ サロゲートモデル未読み込み             │
│   現在は線形近似で表示しています         │
│   より精度の高いPDPを表示するには:       │
│                                          │
│   python generate_surrogates.py \        │
│     --journal result.log                 │
│                                          │

│                                          │

└──────────────────────────────────────────┘
```

### 数理的背景

#### 1変数PDP

```
PDP(x_j) = (1/N) × Σ_i f(x_j, x_{\C j}^{(i)})

x_j     : 対象変数（グリッド上で変化させる）
x_{\C j}: x_j 以外の全変数（実サンプル値を使う）
→ 他変数の影響を周辺化した純粋な x_j の効果を推定
```

#### Individual Conditional Expectation（ICE）

PDPはサンプルを平均化するため個体差が消える。ICEは各サンプルの軌跡を個別表示することで補完する。

```
ICE_i(x_j) = f(x_j, x_{\C j}^{(i)})   ← サンプルiの軌跡
PDP(x_j)   = (1/N) × Σ_i ICE_i(x_j)  ← ICEの平均 = PDP
```

#### 2変数PDP（交互作用の可視化）

```
PDP(x_j, x_k) = (1/N) × Σ_i f(x_j, x_k, x_{\C jk}^{(i)})
→ 2変数のグリッド上でヒートマップとして表示
```

### サロゲートモデル選定

| モデル | 学習速度 | 推論速度 | 精度 | WASM実装 | 備考 |
|---|---|---|---|---|---|
| **線形モデル（Ridge）** | ◎ | ◎ | △ | ◎ | 簡易版・Python不要 |
| **Random Forest** | ◯ | ◎ | ◯ | ✕ | ONNX経由で対応 |
| **Gradient Boosting** | △ | ◎ | ◎ | ✕ | ONNX経由で対応 |
| **Gaussian Process** | ✕（O(n³)） | △ | ◎ | ✕ | 小規模のみ |

**簡易版**: Ridge回帰をRust/WASMで実装（Python不要・即時動作）  
**高精度版**: Random Forest → ONNXエクスポート → ONNX Runtime Webで推論

## 13. 感度分析 / Parameter Importance 設計

### 位置づけと他機能との関係

```
各機能が答える問い:

  感度分析（本章）  →  「どの変数がどの目的にどれだけ効くか？」
                        （変数×目的の寄与行列）
  PDP             →  「どう変化するか？」（形状・応答曲線）
  Scatter Matrix  →  「実際にどう分布しているか？」（実点）
```

OptunaDashboardが持つ重要度はfANOVAベースの**単一目的×全変数の1次元リストのみ**。本ツールでは**4目的×30変数の寄与行列**と**線形モデルによる多変量解析**を提供することが最大の差別化点。

### 分析手法の全体構成：3層設計

```
Layer 1: WASM完結（Python不要・即時）
  - Spearman順位相関係数
  - 標準化偏回帰係数（β）← NEW
  - R²適合度サマリー    ← NEW
  - MIC（最大情報係数、WebWorker非同期）

Layer 2: PDPと兼用（.onnxあり時）
  - Random Forest feature_importance（metadata.jsonから読込）

Layer 3: オプション（shap_values.jsonあり時）
  - SHAP値（目的別・交互作用込み）
```

### 手法の比較

| 手法 | 計算コスト | 交互作用 | 方向性 | Python不要 | 特徴 |
|---|---|---|---|---|---|
| **Spearman相関** | ◎ | ✕ | ◎（±） | ◎ | 非線形単調関係も捕捉 |
| **標準化偏回帰係数β** | ◎ | ✕ | ◎（±） | ◎ | 多重共線性を考慮した純粋効果 |
| **R²適合度** | ◎ | — | — | ◎ | 線形性の度合いを評価 |
| **MIC** | ◯ | ✕ | ✕（0〜1） | ◎ | 非線形・非単調関係も捕捉 |
| **RF重要度** | 即時 | △ | ✕ | △（要.onnx） | PDPと兼用 |
| **SHAP値** | △ | ◎ | ◎（±） | ✕ | 交互作用まで分解可能 |

---

### Layer 1-A: Spearman順位相関係数

```
計算:
  各変数 x_j と各目的 obj_k の間の
  Spearman順位相関係数 ρ(x_j, obj_k) を計算

  ρ = 1 - (6 × Σd²) / (n(n²-1))
  d: 順位の差

特徴:
  - -1〜+1 の値で方向性（増加/減少）も示せる
  - 非線形な単調関係も捕捉可能
  - 外れ値に頑健
  - 変数間の交絡が含まれる点に注意

計算量: O(n log n) × 30変数 × 4目的
目標: 〜500ms (WASM・50000サンプル)
```

### Layer 1-B: 標準化偏回帰係数（β）

Spearmanが1対1の関係しか見ないのに対し、重回帰は**他の変数を固定したときの純粋な独立効果**を示す。

```
モデル:
  obj_k = β1*x1 + β2*x2 + ... + β30*x30 + ε
  ※ 変数・目的ともに標準化（平均0・分散1）してから回帰

βjの解釈:
  「他の変数を固定したとき、x_j が1σ増加すると
   obj_k が β_j σ 変化する」

Spearmanとの差:
  Spearman: x5とobj1の相関=0.82（x12との共線性を含む）
  β:        x5のβ=0.31（x12を固定した純粋効果）
  → 多重共線性の影響を排除した真の寄与を示す

計算:
  Ridge回帰（正則化あり）を採用
  → 30変数の多重共線性に対して安定
  → O(np²) で高速・WASM完結

目標: 〜300ms (WASM・50000サンプル)
```

### Layer 1-C: R²適合度サマリー

線形モデルが各目的をどの程度説明できるかを評価し、**非線形解析への導線**として機能させる。

```
各目的に対してRidge回帰を当てはめ:
  R²(obj1) = 0.81  → 線形性が高い
  R²(obj2) = 0.43  → 非線形解析推奨
  R²(obj3) = 0.67  → 中程度
  R²(obj4) = 0.29  → PDP/SHAPを使用

表示UI:
┌─────────────────────────────────────┐
│ 線形モデル適合度（R²）               │
├──────────┬──────┬────────────────── │
│ 目的     │  R²  │ 解釈              │
├──────────┼──────┼───────────────────│
│ obj1     │ 0.81 │ ✓ 線形性が高い    │
│ obj2     │ 0.43 │ △ 非線形解析推奨  │
│ obj3     │ 0.67 │ ◯ 中程度          │
│ obj4     │ 0.29 │ ✕ PDP/SHAPを推奨 │
└──────────┴──────┴───────────────────┘
R² < 0.5 の目的に自動的に
「PDPまたはSHAP値での分析を推奨します」と表示
```

### Layer 1-D: 残差プロット

線形モデルの残差を可視化し、非線形構造や特異サンプルを発見する。

```
表示:
  予測値（横軸）vs 残差（縦軸）の散布図

判断基準:
  残差がランダム → 線形関係が成立
  残差にパターン → 非線形関係の存在を示唆
    → 「この変数-目的ペアはPDPで詳しく見てください」
       と自動案内

Brushing & Linking連動:
  残差が大きいサンプルをBrushで選択
  → selectedIndices 更新 → 全グラフに反映
  → 「線形モデルで説明できない特異なサンプル」を
     設計空間上（Pareto図・Scatter Matrix）で特定
```

### Layer 1-E: MIC（最大情報係数）

```
特徴:
  - 0〜1 の値（方向性は示さない）
  - 非線形・非単調なあらゆる関数関係を捕捉
  - Spearmanと組み合わせることで相補的に機能
    （SpearmanがMICより低い → 非単調な関係の可能性）

計算:
  O(n × B²)、B = n^0.6 のため重い
  → 10000点にダウンサンプリングして計算
  → WebWorkerで非同期実行

目標: 〜5秒（WebWorker）
```

### Brushing & Linkingとの連動

```
パターン1: 選択サンプルの感度再計算

パターン4: 残差プロット → 特異サンプル特定
```

## 15. 複数Study比較設計

### 前提

**比較対象は同一Journal内のStudyのみ**。複数Journalの読み込みは対象外。同一Journal内なので変数定義は基本的に共通であり、変数の不一致問題はほぼ発生しない。

### 比較のユースケース

```
① アルゴリズム比較（最も一般的）
   同一問題に対して異なるSamplerで実行した結果を比較
   例: study_id=0（TPE） vs study_id=1（NSGA-II）
   → 目的数・変数が同一なので完全比較可能

② 探索設定の変更比較
   変数の探索範囲・制約条件を変えて再実行した結果を比較

③ 試行数の効果確認
   途中経過（10,000試行）と最終（50,000試行）の比較
   → 同一Study内なのでHistoryで対応するほうが自然

④ 目的数違いの横断確認（制限あり）
   単目的 vs 多目的の結果を横断的に確認
   → History・変数分布のみ比較可能
   → Pareto・HV比較は不可・警告表示
```

## 16. ライブ更新設計

### 位置づけ

本ツールの主目的はポストプロセッシング分析だが、**最適化実行中に画面を開いておくと試行追加に合わせて自動更新される**補助機能として提供する。長時間最適化の進捗監視や、良解が出たタイミングでの手動停止判断に活用できる。

### 技術的制約と採用方式

ブラウザのセキュリティモデルにより、通常のFile APIでは読み込み時点のスナップショットしか取得できない。

```
採用方式: File System Access API（FSAPI）+ ポーリング

動作:
  ① ユーザーがライブ更新をONにする
  ② FSAPIでファイルハンドルを取得（ユーザーが許可）
  ③ N秒ごとにファイルサイズを確認
  ④ サイズが増加していれば末尾の差分のみ読み込む
  ⑤ 差分行をJournalパーサに渡しDataFrameに追記
  ⑥ 影響を受けるグラフのみ差分更新

対応ブラウザ: Chrome / Edge
非対応時フォールバック: 手動更新ボタン（ファイル再選択・全件再パース）
```

### ポーリング間隔の設計

```
デフォルト: 5秒
設定可能範囲: 1秒〜30秒

選択基準:
  1秒:  1試行 < 1秒の高速な問題
  5秒:  一般的な用途（デフォルト）
  30秒: 1試行が数分〜数十分かかる問題
```

### 差分パースの設計

Journal は Append-only ログのため差分処理と非常に相性が良い。

```
初回読み込み時:
  ファイルサイズ S0 を記録
  全行をパース → DataFrameを構築

N秒後のポーリング時:
  現在サイズ S1 を取得
  S1 == S0 → 変更なし、スキップ
  S1 >  S0 → バイト位置 S0〜S1 の差分のみ読み込む
            → 差分行をパース → DataFrameに追記
            → S0 = S1 に更新

効果: 50,000試行済みでも新規追加分だけ処理
    → ほぼ一定の処理時間を維持
```

#### 差分パースの注意点

```
注意1: 書き込み途中の行
  最終行のJSONパースが失敗した場合はスキップ
  → 次回ポーリング時に再度読み込む

注意2: RUNNING状態の試行
  差分にRUNNING試行が含まれる場合は保留リストで管理
  → 後続の set_trial_state でCOMPLETE確定後にDataFrameへ追記

注意3: 複数Studyへの書き込み
  差分行に複数study_idが混在する可能性あり
  → 現在選択中のStudyのみ処理
```

## 17. アーティファクト連携設計

### 位置づけ

Optunaのアーティファクト機能（v3.3〜）により、各trialに紐づくCSV・画像・テキスト等のファイルをJournalと連携して分析できる。本ツールではアーティファクトを**設計点の詳細情報**として扱い、グラフ上の点クリックやグループ選択と連動させて成果物を比較・閲覧できるようにする。

### Optunaアーティファクトの仕様

アーティファクトのメタデータ（ファイル名・MIME type・artifact_id）はJournal内の`set_trial_system_attr`に記録され、ファイルの実体はJournalとは独立したディレクトリに保存される。

```jsonl
{"op_code": "set_trial_system_attr",
 "trial_id": 42,
 "key": "artifacts",
 "value": [
   {"artifact_id": "abc123", "filename": "result.png", "mimetype": "image/png"},
   {"artifact_id": "def456", "filename": "data.csv",   "mimetype": "text/csv"}
 ]}
```

### 入力の拡張

```
必須:
  journal.log                ← これまで通り

オプション（アーティファクト連携時）:
  artifacts/                 ← アーティファクトディレクトリ
    abc123                   ← artifact_id がファイル名
    def456
    ...

読み込みUI（ToolBar）:
  [Journal読み込み]  ← 既存
  [Artifactディレクトリ読み込み]  ← 今回追加
  → FSAPIでディレクトリハンドルを取得
  → artifact_idでファイルにアクセス可能になる
```

アーティファクトディレクトリなしでも全機能は通常通り動作する。

### Journalパーサの拡張

```
追加するパース対象:
  set_trial_system_attr（key="artifacts"）
  → trial_id と artifact_id のマッピングを構築

DataFrameに追加される情報:
┌──────────┬──────────────────────────────────────┐
│ trial_id │ artifacts                            │
├──────────┼──────────────────────────────────────┤
```

### PCPのブラシ操作の詳細設計

#### 複数軸AND条件の状態管理

```
filterRanges の構造:
  {
    "x5":  [2.0, 8.0],    ← 軸5のブラシ

WASM側の評価:
  各条件を行ごとにAND評価
  O(n × ブラシ数) → SIMD命令で並列化
  目標: < 5ms（50,000 × 30条件）
```

#### ブラシ条件の保存・命名

```

│ ブラシ条件                            │

│  📌 "低コスト解"   [適用] [削除]      │
└──────────────────────────────────────┘
保存条件はsession.jsonに永続化
レポート生成時に「使用したフィルタ条件」として記録
```

#### 軸の反転（minimize/maximizeへの対応）

```
minimizeの目的軸: 自動反転（上が小さい = 良い方向）
maximizeの目的軸: 通常方向（上が大きい = 良い方向）
→ 全目的軸で「上が良い解」方向に統一
→ 良解のラインが全体的に上に集まる視覚的表現

軸ラベルにdirection表示:
```

## 19. パフォーマンス設計

### 速度目標の全体方針

```
OptunaDashboardとの最大の差別化は「速度」。
以下の3つを動作速度の設計原則とする:

原則1: 初期ロードは段階的に（T+3秒で主要機能が使える）
原則2: インタラクションは常に60fps（ユーザー操作を止めない）
原則3: 分析処理はバックグラウンドで（UIをブロックしない）
```

### 初期ロードの段階的表示設計

```
T+0秒:  ファイル選択ダイアログを閉じた直後

T+0.5秒: Journalメタデータのみパース完了
         → Study一覧UI表示（Study選択可能に）

T+1秒:  選択StudyのPARAMS・VALUESのパース50%完了
         → プログレスバー表示
         → 完了試行分のPareto図を先行描画

T+3秒:  パース100%完了
         → Pareto図・PCP・Bottom Table を先行描画
         → Brushingが有効になる（主要機能が使える）

T+4秒:  Scatter Matrix Mode 2（120セル）描画完了

T+5秒:  感度分析Layer 1（Spearman・β・R²）計算完了
         → 感度ヒートマップ表示

T+5秒〜: MIC・クラスタリング・探索密度KDEをバックグラウンドで順次計算
```

#### Progressive Loadingの実装

```
WebWorker分割パース（3並列）:
  Worker 1: create_study / create_trial（メタデータ）
  Worker 2: set_trial_param（変数値）
  Worker 3: set_trial_value / set_trial_state / user_attr / constraints

  → 完了次第 WASM DataFrameに統合
  → 完了した試行分のみ先行描画可能

プログレス表示:
  ████████░░░░ パース中... 68% (34,000 / 50,000 試行)
```

### WebGLフレームレート目標

| 操作 | 目標 | 許容下限 |
|---|---|---|
| 通常描画（静止時） | 60fps | — |
| グラフ回転・パン・ズーム | 60fps | 30fps |
| Brushingスライダー操作中 | 60fps | 30fps |
| PCPブラシドラッグ中 | 60fps | 30fps |
| クラスタ色更新 | 単発 < 16ms | 33ms |
| Pareto更新（新規trial追加） | 単発 < 16ms | 33ms |

#### 60fps達成のための実装制約

```
制約1: Reactサイクルから完全分離
  WebGLキャンバスはuseEffect後に独自rAFループで動作
  StoreのsubscribeでDOMを経由せずバッファを直接更新

制約2: GPUバッファ更新の最小化
  Brushingフィルタ時: colors[] の alpha値のみ更新
  転送量 = 50,000 × 4bytes = 200KB（十分高速）
  位置・サイズ変更は初期化時のみ

制約3: PCP 30軸・5万ラインの描画
  instanced drawingで全ライン1ドローコール
  ライン座標をGPU側（vertex shader）で計算

制約4: Scatter Matrixテクスチャキャッシュ
  サムネイルはGPUテクスチャとして保持
  再フィルタ時はGPU上でマスクのみ更新
```
